"While building my Athena observability platform, my burn-rate alerts weren't firing. I debugged it systematically — I traced the alert pipeline stage by stage: confirmed the PrometheusRule was loaded with the correct release label, verified the alert expression evaluated true in the Prometheus UI, created a vector(1) always-firing test alert to isolate whether the problem was my expression or the delivery pipeline, checked the Prometheus-to-Alertmanager connection, and finally found service and labels are not matching that's why unable to load the data in prometheus. The lesson was that with multi-stage pipelines, I isolate by finding the last working stage and debugging the next one."

"My custom burn-rate alerts weren't firing. Rather than assume the pipeline was broken, I isolated it: I deployed a vector(1) always-firing test alert and confirmed it reached webhook.site, which proved the entire Prometheus → Alertmanager → webhook chain worked. I also saw the built-in etcd alerts being delivered. That narrowed the problem to my specific rule — either a label mismatch preventing rule load, or the SLI expression not crossing threshold because the service was healthy. The root cause was that everything is good but there's mistake in rules.yaml file once I fixed the SLO logic now its delivered. The lesson: when a multi-stage pipeline 'doesn't work,' inject a known-good signal to find which stage actually fails, instead of guessing."
